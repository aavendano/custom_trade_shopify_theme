
# **Guía para el Programador – Arquitectura Headless Híbrida PLT**

Esta guía explica **cómo funciona el sistema**, **cómo se estructura el código** y **qué responsabilidades tiene cada tecnología** dentro de la tienda PlayLoveToys.
El objetivo es asegurar que cualquier programador pueda extender la tienda sin romper SEO, performance o diseño.

---

## **1. Arquitectura General**

El frontend de PlayLoveToys combina tres sistemas:

### Bulma (con prefijo `b-`)

Proporciona el layout completo: columnas, espaciado, grid, botones, tipografía y helpers visuales.
Todo el HTML debe seguir esta convención para mantener consistencia.

### Alpine.js

Es el sistema reactivo ligero. Maneja interacciones como:

* Quick View
* cambios de layout (horizontal/vertical en mobile)
* eventos del carrito
* modales
* estados locales
* lógica para sliders o toggles
* stores globales simples

No hay virtual DOM ni build: Alpine se escribe directamente en HTML.

### Shopify Storefront Web Components

Son la capa de **datos dinámicos**:

* `<shopify-context>`
* `<shopify-list-context>`
* `<shopify-data>`
* `<shopify-money>`
* `<shopify-media>`
* `<shopify-cart>`
* `<shopify-variant-selector>`

Estos componentes reemplazan la necesidad de usar Liquid para cargar productos, precios y variantes en tiempo real.

La tienda sigue siendo un theme Shopify, pero se comporta como un micro-frontend moderno.

---

## **2. Reglas de Oro para el Desarrollo**

### Mantener SEO server-side

Cada página debe incluir HTML renderizado vía Liquid:

* `<h1>`
* `<img>`
* títulos de productos
* enlaces a productos
* descripción corta

Los Web Components **nunca deben ser la única fuente de contenido**.
Siempre deben enriquecer, no reemplazar.

### Mantener performance

* Cargar Web Components en `theme.liquid` con `type="module"`.
* Limitar lógica Alpine al DOM necesario.
* Usar imágenes cuadradas con `object-fit: contain` para estabilidad visual.
* Mantener Bulma purgado usando tu sistema actual.

### Mantener responsividad móvil

La tienda es mobile-first por diseño.
Las cards deben:

* ser horizontales en mobile (imagen izquierda + texto derecha),
* verticales en desktop.

Usar Alpine para detectar ancho:

```html
x-data="{ isMobile: window.innerWidth < 769 }"
x-init="window.addEventListener('resize', () => isMobile = window.innerWidth < 769)"
```

## **3. Componentes Esenciales del Sistema**

### Contenedores con overlays (PLT custom)

Utilizar:

* `b-has-image-overlay`
* `b-tag-bottom-right`
* `b-badge-pill-primary` (colores generados dinámicamente a partir del mapa `bulma.$colors`)

Estos helpers viven en `src/bulma/custom/_overlays.scss`.

### Product Card PLT

Toda card de producto debe seguir el patrón:

* Layout Bulma
* Imagen con `<shopify-media>`
* Título con `<shopify-data>`
* Precio con `<shopify-money>`
* Badge dinámico con `shopify-attr--hidden`
* Quick View conectado vía Alpine

El card debe funcionar en:

* Colecciones
* Recomendaciones
* Quick View
* Secciones personalizadas
* Páginas estáticas que contengan productos

Esto permite tener **un único componente reutilizable**.

---

## **4. Uso Correcto de Shopify Web Components**

### Cargar una colección

```html
<shopify-context type="collection" handle="{{ collection.handle }}">
  <template>
    <shopify-list-context type="product" query="collection.products" first="24">
      <template>
        <!-- Aquí va la card PLT -->
      </template>
    </shopify-list-context>
  </template>
</shopify-context>
```

### Dentro de la card

```html
<shopify-media query="product.selectedOrFirstAvailableVariant.image"></shopify-media>

<shopify-data query="product.title"></shopify-data>

<shopify-money query="product.selectedOrFirstAvailableVariant.price"></shopify-money>
```

### Condicionales

```html
shopify-attr--hidden="!(product.selectedOrFirstAvailableVariant.compareAtPrice > product.selectedOrFirstAvailableVariant.price)"
```

### Quick View

```html
@click="
  $refs.modalContext.update(event);
  document.getElementById('plt-product-modal').showModal();
"
```

## **5. Alpine como sistema reactivo**

### Alpine controla:

* Eventos de usuario
* Estados locales
* Estados globales simples
* Apertura/cierre de modales
* Cambios de layout según mobile/desktop
* Lógica ligera de UI

### No debe controlar:

* Carga de datos (esto lo hacen los Web Components)
* HTML complejo o plantillas grandes (usar liquid para lo base)
* Reemplazar estructura de producto o elementos SEO

Alpine no debe ser usado como framework completo; su función es ser una **capa de comportamiento**.

---

## **6. Flujo recomendado para construir nuevas secciones**

1. Crear el HTML con Bulma primero.
2. Añadir estructura compatible con SEO usando Liquid.
3. Insertar Web Components dentro de esa estructura.
4. Añadir control de interacción con Alpine.
5. Probar mobile-first.
6. Validar Lighthouse (PLT tiene métricas estrictas).
7. Revisar que las clases Bulma purguen correctamente.

El orden es importante:
Liquid → Bulma → Web Components → Alpine.

---

## **7. Errores que deben evitarse**

* Renderizar productos **solo** vía Web Components (rompe SEO).
* Omitir `<a href="{{ product.url }}">` en cards (rompe crawl).
* Estilar contenido interno del Shadow DOM (no funciona).
* Usar lógica de negocio en JavaScript en vez de Web Components.
* Colocar Alpine como controlador de datos (evitar fetch manual).
* Añadir múltiples scripts de Web Components (solo uno en theme.liquid).
* Usar clases Bulma sin prefijo `b-` (no serán purgadas correctamente).
* Crear CSS inline repetitivo (desacoplar a SCSS).

---

## **8. Objetivo final de este sistema**

El programador debe entender que la meta NO es hacer un headless puro.
La meta es:

* Mantener SEO perfecto gracias al HTML Liquid.
* Mantener velocidad gracias a Bulma + Web Components.
* Mantener interacción moderna gracias a Alpine.
* Mantener consistencia visual gracias a PLT-Design-System.
* Mantener el checkout nativo de Shopify.
* Mantener compatibilidad con apps.

Es una arquitectura **moderna, ligera, escalable y mobile-first**, pero sigue siendo un theme Shopify.

---

## **Conclusión**

El programador que trabaje con esta arquitectura debe entender que:

**PlayLoveToys no es headless.
Tampoco es un theme tradicional.
Es un híbrido optimizado para SEO, velocidad, diseño y facilidad de desarrollo.**

* Bulma = layout
* Alpine = interacción
* Web Components = datos
* Liquid = base SEO y SSR

Todo lo demás gira alrededor de esta idea.
